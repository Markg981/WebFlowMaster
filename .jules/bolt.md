## 2026-03-30 - Playwright BrowserPool Resource Leaks
**Learning:** Reusing Playwright Browser instances via `browser-pool.ts` prevents process churn and speeds up requests, but it shifts the responsibility of cleanup. A single `Browser` can hold multiple isolated contexts. If a pooled browser is simply released back into the pool without first explicitly calling `await context.close()` and `await page.close()`, the pages remain active in the background, leading to a massive memory leak and OOM crash.
**Action:** When implementing or refactoring methods in `playwright-service.ts` to use `browserPool.acquire()`, ALWAYS ensure that `await page.close().catch(() => {})` and `await context.close().catch(() => {})` are executed in a `finally` block before calling `await pool.release(browser)`.
